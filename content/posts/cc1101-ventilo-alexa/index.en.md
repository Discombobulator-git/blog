Some projects look trivial on paper. This one was: capture the radio signal of a ceiling fan remote, replay it with a CC1101 module driven from a Raspberry Pi, then wire the whole thing to Alexa for voice control. Three lines of spec. Several weeks of reverse engineering.

Here's the full story, with the dead ends, the real blockers, and the code that runs today in production in the living room.

## The hardware

A **CC1101** module (433.92 MHz, OOK modulation) connected over SPI to a **Raspberry Pi 5**, nicknamed RaspiLite. Standard wiring: CS on GPIO8, CLK on GPIO11, MOSI on GPIO10, MISO on GPIO9. The GDO0 pin, which carries the raw OOK signal, was originally meant to go on GPIO25, but that pin turned out to be damaged on the Pi. Moved to GPIO24 instead.

Two tools for observing and capturing the signals: an RTL-SDR dongle paired with GQRX to watch the spectrum live, and a Kingst LA1010 logic analyzer to precisely capture the frames sent by the real remote.

## First trap: the Raspberry Pi 5 changes the rules

The first obstacle showed up before RF was even in the picture. On the Pi 5, the RP1 chip that handles GPIO fundamentally changes how the hardware is accessed. Result: `pigpio`, the reference library for this kind of precise timing, no longer works, since it depends on direct access to BCM registers that no longer exist in the same form.

Second disappointment: `lgpio.tx_wave()`, supposed to generate a precise hardware PWM signal, turned out to be broken too. On the SDR, the levels never switched after the very first pulse.

The solution that stuck was more hands-on: manual **bit-banging**, using `lgpio.gpio_write()` driven by a busy-wait loop based on `time.perf_counter()`. Not `time.sleep()`, which has far too much jitter on a non-real-time Linux for 300 and 700 microsecond pulses. The measured result was near-perfect timing.

```python
def busy_wait(seconds):
    """Attente active haute precision (time.sleep() a trop de jitter sous Linux
    non temps-reel pour des impulsions de 300/700us)."""
    target = time.perf_counter() + seconds
    while time.perf_counter() < target:
        pass
```

## Noise we inflicted on ourselves

While listening, the RX capture was drowned in parasitic noise. A simple count of detected edges revealed something unexpected: about 467 edges per second with the Pi disconnected from the GDO0 wire, versus 1055 per second once connected. Simply connecting the wire more than doubled the electromagnetic noise, generated by the Pi itself.

First reflex adopted: systematically disconnect the Pi from the GDO0 wire during pure capture phases.

The real fix came next, on the CC1101 configuration side: reduce the maximum allowed RX gain via the `AGCCTRL2` register, going from `0x43` to `0x63`, roughly a 9 dB reduction in maximum LNA gain. Captures went from "full of noise-frames" to 100% clean frames.

## A counter hidden in the suffix

Each remote button sends a 32-bit frame. Comparing several captures of the same button, a pattern emerged: the first 25 to 26 bits stay perfectly stable from one press to the next. These identify the button. But the last 6 to 8 bits change on every new press, while staying consistent across repetitions of the same press.

A typical signature of an anti-replay counter, a lightweight rolling code. Concrete consequence: old codes previously saved in the script had become stale on this suffix, while the prefix remained usable as-is.

## The real blocker: an invisible bit

This is where the project truly got stuck. Freshly recaptured codes, a signal visibly clean on GQRX, and yet nothing happened on the actual fan. Even tested at point-blank range, which at least ruled out a range or antenna gain hypothesis, and refocused the investigation on the frame structure itself rather than transmit power.

Going back to the raw CSV files from the logic analyzer by hand, working around a bug in the automatic decoding script, the cause emerged: each frame is actually **33 bits, not 32**. The 33rd bit is a clean high pulse that strictly alternates 0, 1, 0, 1 on each repetition within the same button press, an identical behavior confirmed across the three buttons tested.

The decoding script was losing this bit because its low portion electrically merged with the silence between two frames, with no detectable transition. The 32 stored bits were therefore correct, but the transmitted frame was structurally too short, and the fan's receiver silently rejected it.

The fix: add a toggled end bit, starting at `0` then alternating on each repetition, right after the 32 code bits and before the inter-frame silence.

```python
def bit_banging_tx(h, bits, repeats):
    """Chaque trame reelle fait 33 bits, pas 32 : un bit supplementaire
    termine la trame, perdu par le script d'analyse car sa portion basse
    fusionne avec le silence inter-trame. Ce bit alterne 0/1/0/1 a chaque
    repetition sur toutes les captures observees."""
    toggle_bit = '0'
    for r in range(repeats):
        for bit in bits + toggle_bit:
            if bit == '1':
                lgpio.gpio_write(h, GDO0_PIN, 1)
                busy_wait(LONG_US / 1e6)
                lgpio.gpio_write(h, GDO0_PIN, 0)
                busy_wait(SHORT_US / 1e6)
            else:
                lgpio.gpio_write(h, GDO0_PIN, 1)
                busy_wait(SHORT_US / 1e6)
                lgpio.gpio_write(h, GDO0_PIN, 0)
                busy_wait(LONG_US / 1e6)
        lgpio.gpio_write(h, GDO0_PIN, 0)
        time.sleep(INTERFRAME_GAP_US / 1e6)
        toggle_bit = '1' if toggle_bit == '0' else '0'
```

The result didn't take long. The `onoff` button made the real fan react (beep, state change) right after this fix. Then `lumiere`, `v2` and `v1` followed, confirmed working with exactly the same generic fix, with no button-specific handling.

Other, more minor adjustments came along with this breakthrough: the `IOCFG0` register corrected from `0x2E` (high impedance) to `0x2D` (asynchronous TX data input), the addition of an `SCAL` synthesizer calibration strobe before `STX` which cleaned up a signal previously seen as dirty and wideband on GQRX, and a `PATABLE` value readjusted from `0xC0` to `0xC6`.

## Wiring it all up to Alexa

The final goal remained: voice control. Two options were on the table. A custom Alexa Skill hosted on AWS Lambda, ruled out as heavier to maintain. Or **Home Assistant in Docker** paired with the `emulated_hue` component, which makes Home Assistant look like a Philips Hue bridge to Alexa, with local discovery and no paid cloud account. This second option was the one chosen, the simplest thing that works.

The complete chain put in place:

1. A Home Assistant Docker container.
2. A small native HTTP server, `ventilo_bridge.py`, running outside Docker to keep direct access to SPI and GPIO, exposing one endpoint per button and launching the transmit script as a subprocess.
3. In the Home Assistant configuration, `rest_command`s that call this server, and optimistic template switches that call these `rest_command`s.
4. `emulated_hue` configured to expose these switches as virtual Hue bulbs.

```python
class Handler(BaseHTTPRequestHandler):
    def _handle(self):
        bouton = self.path.strip("/")
        if bouton not in BOUTONS_EXPOSES:
            self.send_response(404)
            self.end_headers()
            self.wfile.write(f"Bouton inconnu: {bouton}\n".encode())
            return
        try:
            result = subprocess.run(
                [sys.executable, SCRIPT, bouton],
                capture_output=True, text=True, timeout=15,
            )
        except subprocess.TimeoutExpired:
            self.send_response(504)
            self.end_headers()
            self.wfile.write(b"Timeout pendant la transmission\n")
            return
        if result.returncode == 0:
            self.send_response(200)
            self.end_headers()
            self.wfile.write(f"OK: {bouton}\n".encode())
        else:
            self.send_response(500)
            self.end_headers()
            self.wfile.write(result.stderr.encode())
```

One more trap was waiting around the corner. With `expose_by_default: false`, listing an entity under `entities:` with just a `name:` isn't enough to actually expose it to the Hue API. The API returned `{}` even though the switches did exist on the Home Assistant side. You have to explicitly add `hidden: false` under each entity. The alternative would have been to set `expose_by_default: true`, but that exposes absolutely everything, including about twenty unrelated entities like media switches.

```yaml
emulated_hue:
  host_ip: 192.168.1.37
  listen_port: 80
  expose_by_default: false
  entities:
    switch.ventilo:
      name: Ventilo
    switch.lumiere_ventilo:
      name: Lumiere Ventilo
```

The end-to-end test was done with a `curl` call reproducing exactly what Alexa would send, a `PUT` on `/api/anyuser/lights/1/state`. The fan reacted. The complete chain, from the Hue API to the CC1101 through Home Assistant and the bridge server, was validated.

A `crontab -e @reboot` automatically restarts the bridge server on boot. Only one step remains that can't be automated: manually discovering the devices from the Alexa mobile app.

## Where things stand, and what's next

At this stage, the complete chain works end to end: voice command or Hue API, Home Assistant switch, `rest_command`, Python bridge server, CC1101 in bit-banging mode, RF reception by the fan. The four buttons `onoff`, `lumiere`, `v1` and `v2` are operational.

The next step, decided but not yet started, is porting the transmit side to an already-ordered **ESP32-WROOM-32**. The idea is to get away from the Pi 5's timing issues, which force manual bit-banging, thanks to the ESP32's hardware RMT peripheral, designed precisely to generate precisely-timed signals natively. This would also open the door to native ESPHome integration in Home Assistant, without the `emulated_hue` detour.

## Key takeaways

This project is a good illustration of a classic RF reverse engineering principle: a signal can look clean on an oscilloscope or SDR while still being structurally incomplete. A bit invisible to the analysis tool, because it blends into the inter-frame silence, can be enough to block 100% of receptions on the recipient's side.

And sometimes, the simplest solution that works (an `emulated_hue` component rather than a cloud Alexa Skill) is worth more than an infrastructure that's more "elegant" on paper but heavier to maintain.

## Update: porting to a Raspberry Pi Zero W

Rather than the ESP32 initially planned, the transmit side ended up being ported to a **Raspberry Pi Zero W** instead. On this classic BCM2835 chip (without the Pi 5's RP1), `pigpio` works natively: manual bit-banging was replaced with wave-DMA signal generation via `pigpio.wave_send_once()`, with microsecond-accurate timing guaranteed and no blocking busy-wait loop.

The script now handles nine commands (`onoff`, `lumiere`, `v1` through `v6`, `inversion`), and the HTTP bridge was rewritten in Flask (`ventilo_api.py`) rather than as a hand-rolled `BaseHTTPRequestHandler` server.
